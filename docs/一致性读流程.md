# 一致性读流程

本文档梳理当前项目中 `Get` 请求的线性一致读实现。当前实现采用 **ReadIndex 风格的一致性读**：读请求不再写入 Raft 日志，而是由 Leader 先向多数派发送一轮空 `AppendEntries` 心跳，确认自己仍然拥有 Leader 地位，再等待本机 KV 状态机追到本次读取对应的 `readIndex`，最后直接读取本地 SkipList。

这种方式相比“读也写日志”的方案更轻：读请求不会膨胀 Raft log，也不需要每次读都复制一条 `Get` 日志。

## 一、整体架构

```text
Client / Clerk
    |
    | Get RPC
    v
KvServer
    |
    | ReadIndex()
    v
Raft Leader
    |
    | empty AppendEntries heartbeat
    v
Raft peers majority
    |
    | confirm current leader term
    v
KvServer waits local apply >= readIndex
    |
    | local read
    v
SkipList KV state machine
```

项目中相关分层如下：

| 层 | 目录/文件 | 职责 |
| --- | --- | --- |
| 客户端层 | `src/raftClerk/clerk.cpp` | 生成 `ClientId`、递增 `RequestId`，向最近的 Leader 发送请求，失败后轮询重试 |
| KV 服务层 | `src/raftCore/kvServer.cpp` | 处理 `Get/Put/Append`，读请求走 `ReadIndex` 后本地读，写请求仍走 Raft 日志 |
| Raft 共识层 | `src/raftCore/raft.cpp` | 选举、日志复制、提交推进、快照安装、读前 Leader 确认 |
| RPC 协议层 | `src/raftRpcPro/*.proto` | 定义 KV RPC 和 Raft RPC 消息 |
| 状态机存储层 | `src/skipList/include/skipList.h` | 保存最终 KV 数据 |

## 二、核心数据结构

### 1. Clerk 请求标识

`Clerk` 为每个客户端生成唯一 `m_clientId`，并为每次请求递增 `m_requestId`。

```cpp
std::string m_clientId;
int m_requestId;
int m_recentLeaderId;
```

`GetArgs` 会携带：

```proto
message GetArgs {
  bytes Key = 1;
  bytes ClientId = 2;
  int32 RequestId = 3;
}
```

`ClientId + RequestId` 用于识别客户端请求。当前 `Get` 不再进入 Raft 日志，但本地读取成功后仍会更新 `m_lastRequestId`，保证快照中的客户端请求进度继续单调。

### 2. Raft::ReadIndex

Raft 新增接口：

```cpp
bool Raft::ReadIndex(int* readIndex);
```

它负责读前确认：

1. 加锁检查当前节点是否仍是 `Leader`。
2. 记录当前 `term` 和 `commitIndex`，其中 `commitIndex` 作为本次读的 `readIndex`。
3. 向其他节点发送一轮不带日志条目的 `AppendEntries`。
4. 如果收到多数派成功响应，返回 `true` 和 `readIndex`。
5. 如果发现更高 term，立即转为 `Follower` 并返回失败。

`readIndex` 的含义是：本次读必须至少看到该 index 之前已经提交的全部写入。

### 3. KV 层 apply 进度

KVServer 新增：

```cpp
int m_lastAppliedRaftLogIndex;
```

Raft 自己的 `m_lastApplied` 只表示日志已经被 Raft applier 推送为 `ApplyMsg`，但不代表 KV 状态机一定执行完。因此 KV 层需要单独记录状态机真正应用到的日志下标。

`GetCommandFromRaft` 在处理完一条 committed log 后更新：

```cpp
m_lastAppliedRaftLogIndex = std::max(m_lastAppliedRaftLogIndex,
                                     message.CommandIndex);
```

快照安装后也会更新：

```cpp
m_lastAppliedRaftLogIndex = std::max(m_lastAppliedRaftLogIndex,
                                     message.SnapshotIndex);
```

### 4. m_lastRequestId

KVServer 继续维护：

```cpp
std::unordered_map<std::string, int> m_lastRequestId;
```

它主要用于 `Put/Append` 去重，防止客户端 RPC 重试导致重复写。`Get` 是只读操作，重复执行不会改变业务数据，但读取成功后会更新该表，让客户端请求进度在快照恢复后保持一致。

## 三、一致性读完整流程

### 1. Clerk 发送 Get

入口：`src/raftClerk/clerk.cpp`

```cpp
std::string Clerk::Get(std::string key)
```

Clerk 构造 `GetArgs`：

```cpp
args.set_key(key);
args.set_clientid(m_clientId);
args.set_requestid(requestId);
```

如果 RPC 失败，或者服务端返回 `ErrWrongLeader`，Clerk 会换下一个节点重试。

### 2. KvServer 构造本地读 Op

入口：`src/raftCore/kvServer.cpp`

```cpp
void KvServer::Get(const raftKVRpcProctoc::GetArgs *args,
                   raftKVRpcProctoc::GetReply *reply)
```

服务端仍会构造一个 `Op`，但这个 `Op` 不再提交给 Raft 日志，只用于本地状态机读取和请求进度记录：

```cpp
op.Operation = "Get";
op.Key = args->key();
op.Value = "";
op.ClientId = args->clientid();
op.RequestId = args->requestid();
```

### 3. 通过心跳确认 Leader 地位

KVServer 调用：

```cpp
int readIndex = -1;
if (!m_raftNode->ReadIndex(&readIndex)) {
  reply->set_err(ErrWrongLeader);
  return;
}
```

`ReadIndex` 会向多数派发送空 `AppendEntries`。如果多数派在当前 term 下承认该节点为 Leader，说明这个节点仍然拥有服务线性一致读的资格。

### 4. 等待本地状态机追上 readIndex

拿到 `readIndex` 后，KVServer 不能立刻读。原因是 Raft 的 `commitIndex` 已经推进，不代表 KV 状态机一定 apply 完。

因此 `Get` 会等待：

```cpp
WaitApplied(readIndex)
```

内部检查：

```cpp
m_lastAppliedRaftLogIndex >= readIndex
```

如果超时，则返回 `ErrWrongLeader`，让 Clerk 换节点重试。

### 5. 本地读取 SkipList

确认当前节点仍是 Leader，并且本地状态机已经追到 `readIndex` 后，才能读本地 SkipList：

```cpp
ExecuteGetOpOnKVDB(op, &value, &exist);
```

读取成功返回：

```cpp
OK + value
```

不存在则返回：

```cpp
ErrNoKey
```

## 四、为什么这样能保证线性一致

这个读路径依赖三个条件：

1. **读前确认 Leader**

   Leader 必须向多数派发送一轮空 `AppendEntries` 并收到成功响应。旧 Leader 如果已经失去多数派，就无法通过这一步。

2. **读取位置绑定 commitIndex**

   `ReadIndex` 返回的是发起读时 Leader 已知的 `commitIndex`。本次读至少要看到这个 index 之前已经提交的所有写入。

3. **本地状态机追到 readIndex**

   KVServer 不只相信 Raft 的 `commitIndex`，还要等 `m_lastAppliedRaftLogIndex >= readIndex`，确保本地 SkipList 已经应用了这些写入。

因此，读请求不会从旧 Leader 或落后状态机读到旧值，同时又避免了为每次 `Get` 追加 Raft 日志。

## 五、与写请求的关系

`Put/Append` 仍然走原来的日志复制流程：

```text
Put/Append -> Raft::Start -> AppendEntries -> majority commit -> apply -> reply OK
```

`Get` 走新的只读流程：

```text
Get -> ReadIndex heartbeat quorum -> wait local apply -> local SkipList read
```

也就是说，只有写请求进入 Raft log；读请求只使用 Raft 心跳作为 Leader 身份确认和读屏障。

## 六、后续可优化点

当前实现是保守版 ReadIndex：每次 `Get` 都同步发一轮心跳确认。后续可以继续优化：

| 优化 | 说明 |
| --- | --- |
| 批量 ReadIndex | 多个并发读共享同一轮心跳确认 |
| Leader Lease | 在租约有效期内跳过每次读的心跳确认，直接本地读 |

## 七、Leader 重定向

Raft 节点会维护最近一次确认到的 Leader：

```cpp
int m_leaderId;
```

Follower 在收到合法 `AppendEntries` 时记录 `args->leaderid()`；节点自己成为 Leader 时记录 `m_me`。当 KVServer 需要返回 `ErrWrongLeader` 时，会把已知 Leader 编码到现有 `Err` 字段里：

```text
ErrWrongLeader:<leaderId>
```

Clerk 收到后会优先跳转到该 `leaderId`，如果没有携带 Leader 信息，才退回原来的轮询下一个节点。

## 八、读指标

KVServer 对 ReadIndex 读路径做了轻量统计：

| 指标 | 含义 |
| --- | --- |
| `readIndexOk` | ReadIndex 心跳确认成功次数 |
| `readIndexFail` | ReadIndex 心跳确认失败次数 |
| `applyTimeout` | 本地状态机等待 `readIndex` 超时次数 |
| `avgReadIndexMs` | 平均 ReadIndex 心跳确认耗时 |
| `avgApplyWaitMs` | 平均本地 apply 等待耗时 |

每累计 100 次读请求，会输出一行：

```text
[ReadMetrics] server=0 total=100 readIndexOk=100 readIndexFail=0 applyTimeout=0 avgReadIndexMs=2 avgApplyWaitMs=0
```
